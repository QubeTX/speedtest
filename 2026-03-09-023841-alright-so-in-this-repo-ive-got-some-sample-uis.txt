     QubeTX Speed Test — Implementation Plan

     Context

     The repo has 4 independent sample UI React components (generated by another
      AI) in sample-ui/ with no build system, no routing, no shared state, and
     no actual speed test functionality. The goal is to turn these into a fully
     functional, responsive speed test web app branded by QubeTX, using two
     browser-native speed test providers (Cloudflare and M-Lab/NDT7).

     Key problems to solve:
     - Views are disconnected — unify into a single app with shared state
     - Tape reel animation needs to be tied to actual test phases (spin when
     testing, stop when idle/complete)
     - "TEST COMPLETE" stamp overlaps "RUN AGAIN" button — fix z-layering
     - Fixed dimensions (1440x900) in some views — make fully responsive
     - No real speed test logic — integrate Cloudflare and NDT7 providers
     - Settings page needs only 2 providers (not 6 servers)
     - Need QubeTX/SHAUGHV branding throughout

     ---
     Tech Stack

     - Vite + React 18 + TypeScript
     - React Router v7 (already partially used in samples)
     - React Context for state management (minimal deps)
     - CSS-in-JS (inline styles, matching existing sample pattern) + global
     keyframes in index.css
     - @cloudflare/speedtest — download, upload, latency, jitter, packet loss
     - @m-lab/ndt7 — download, upload, latency (+ derived jitter)

     ---
     Speed Test Provider Abstraction

     Interface (src/types/speedtest.ts)

     type TestPhase = 'idle' | 'discovering' | 'latency' | 'download' | 'upload'
      | 'complete' | 'error';
     type ProviderMode = 'both' | 'cloudflare' | 'ndt7';
     type TestDuration = 'auto' | 15 | 30 | 60 | 120 | 300 | 600; // seconds

     interface SpeedTestProgress {
       phase: TestPhase;
       currentProvider: string;       // Which provider is currently running
       ping: number | null;           // ms
       jitter: number | null;         // ms
       downloadSpeed: number | null;  // Mbps
       uploadSpeed: number | null;    // Mbps
       packetLoss: number | null;     // 0-1 ratio (Cloudflare only)
       downloadProgress: number;      // 0-100
       uploadProgress: number;        // 0-100
       serverName: string | null;
       error: string | null;
     }

     interface SpeedTestResult {
       provider: string;              // 'cloudflare' | 'ndt7' | 'aggregated'
       ping: number;
       jitter: number;
       downloadSpeed: number;
       uploadSpeed: number;
       packetLoss: number | null;
       serverName: string;
       timestamp: number;
       // Per-provider breakdown (populated in aggregated mode)
       providerResults?: {
         cloudflare?: SpeedTestResult;
         ndt7?: SpeedTestResult;
       };
       isp?: string;                  // ISP name from Cloudflare metadata
     }

     interface SpeedTestProvider {
       name: string;
       supportsPacketLoss: boolean;
       requiresConsent: boolean;
       start(onProgress: (p: SpeedTestProgress) => void, duration?:
     TestDuration): Promise<SpeedTestResult>;
       stop(): void;
     }

     interface Settings {
       providerMode: ProviderMode;      // Default: 'both'
       testDuration: TestDuration;      // Default: 'auto'
       dataPolicyAccepted: boolean;     // Required when NDT7 involved
       speedUnit: 'auto' | 'Mbps' | 'Kbps' | 'Gbps';
       autoCopyResults: boolean;
       soundEffects: boolean;
     }

     Aggregated Mode Logic (useSpeedTest hook)

     When providerMode === 'both':
     1. Run Cloudflare test first → collect results
     2. Run NDT7 test second → collect results
     3. Average shared metrics: avg_dl = (cf_dl + ndt_dl) / 2, same for UL, ping
     4. Jitter: use Cloudflare's (more accurate), fall back to NDT7 derived
     5. Packet loss: use Cloudflare's (NDT7 doesn't provide)
     6. ISP: use Cloudflare's metadata
     7. Final result has provider: 'aggregated' with providerResults breakdown
     8. UI shows averaged values prominently, with per-provider values in
     smaller text
     9. Progress: 0-50% maps to Cloudflare phases, 50-100% maps to NDT7 phases
     10. Status text updates: "TESTING VIA CLOUDFLARE" → "TESTING VIA M-LAB"

     Cloudflare Adapter

     - new SpeedTest({ autoStart: false }) → onResultsChange for live progress
     - results.getDownloadBandwidth() (bps → Mbps), getUploadBandwidth(),
     getUnloadedLatency(), getUnloadedJitter(), getPacketLoss()
     - onFinish resolves promise, onError rejects
     - No server selection (always Cloudflare edge)

     NDT7 Adapter

     - ndt7.test(config, callbacks) — sequential download then upload
     - Worker files must be in public/ (postinstall script copies them)
     - config.userAcceptedDataPolicy = true required (enforced in settings)
     - Latency from TCPInfo.MinRTT (µs → ms), jitter derived from consecutive
     RTT samples
     - serverChosen callback provides server info

     ---
     WiFi / Network Info Display

     What's available in browsers (navigator.connection / Network Information 
     API):

     - type: wifi, cellular, ethernet, none
     - effectiveType: 4g, 3g, 2g, slow-2g
     - downlink: Estimated bandwidth (Mbps)
     - rtt: Estimated round-trip time (ms)

     What's NOT available in browsers (requires native/OS APIs):

     - WiFi SSID name
     - WiFi provider / ISP name
     - Cell carrier name

     Implementation (useNetworkInfo hook + NetworkBadge component):

     - Display connection type icon (WiFi/Cellular/Ethernet) in SysInfo area
     - Show effective type (e.g., "WiFi • 4G") as a badge
     - Listen for change events on navigator.connection to update live
     - Falls back gracefully if API unavailable (most Chromium browsers support
     it, Firefox/Safari partial)
     - ISP name: Can potentially get from Cloudflare's test metadata (Cloudflare
      returns the ISP/ASN info) — display if available
